## Corrections — where the agent went wrong and how it was caught

1. **Decision-process ordering bug (caught by me, 2026-07-14).** The agent's architecture decision
   sequence placed the mental model (behaviour layers) at step 9 — *after* the composition decision
   it was supposed to inform, and RG enforcement *after* the ranking model even though the
   eligibility pre-filter defines the candidate pool. I challenged the ordering; the agent confirmed
   both dependency violations and re-sequenced (mental model → composition; RG placement → candidate
   generation).

2. **Letter-prefix jargon (caught by me).** The agent introduced Y/Z-market/Z-session/W labels for
   behaviour layers — opaque shorthand that would not survive a handoff. Asked for descriptive names
   and a table; the agent replaced all references.

3. **Over-compressed ADR prose with undefined jargon (caught by me, 2026-07-14).** The first
   accepted ADRs were written in dense fragment chains, and used the invented term "rule pack"
   without defining it. Flagged as hard to follow; the agent rewrote
   [ADR-0001](../adr/0001-offline-nearline-online-composition.md)/[0005](../adr/0005-rg-enforcement-point.md)/[0006](../adr/0006-multi-tenancy.md)/[0007](../adr/0007-cost-model.md) in
   plain prose, created GLOSSARY.md for all coined terms, and added the style rule to CLAUDE.md so
   later sessions inherit it. Decision content was unchanged — this was purely a readability defect.

4. **Missing de-duplication policy in candidate generation (caught by me, 2026-07-14).** The
   agent's blend design let the same candidate arrive from several sources without saying what
   happens next. The fix mattered beyond hygiene: provenance became a ranking feature, and the
   promotional tag was made merge-proof — an RG edge case (an item arriving both organically and
   as a promotion must never lose its marketing flag) that the agent had not considered.

5. **Under-specified areas surfaced by ADR review (caught by me, 2026-07-14).** Four gaps in the
   accepted ADRs: (a) nearline's mechanism and benefits were asserted but never explained, and
   "which users get recomputed" had no policy — [ADR-0001](../adr/0001-offline-nearline-online-composition.md) gained a concrete explainer plus a
   three-tier targeting policy with a bounded worker budget; (b) cross-type score comparability
   appeared only deep in [ADR-0003](../adr/0003-ranking-model.md) though it is a system requirement — promoted to TASKS §1 and
   the glossary; (c) per-prediction feature attributions were proposed as if logged on every
   serve — challenged as unrealistic, redesigned as attribution-on-replay with honest limits
   stated (explains model arithmetic, not causality; not the compliance story); (d) the label
   design stopped at a single signal — expanded to a signal ladder with an explicit
   forbidden-signals list, multi-objective composition at the ordering stage with inspectable
   weights, and a permanent holdout cohort so long-term effects are measured before anything
   optimises for them.

6. **Flawed training-data design, caught by two pointed questions (2026-07-14).** The agent
   proposed training the ranker on organic positives plus recsys impressions, with "engagement
   surface" as a training feature. Asked how propensity scoring works for non-recsys surfaces,
   and flagged the surface feature as suspect. Both objections held: organic exposure has no
   logged propensity (so those rows can never support counterfactual evaluation, and the agent's
   "exposure-independent" claim was overstated — organic data carries the operator UI's own
   bias), and the surface feature correlated exactly with the negative-sampling method, meaning
   it would have absorbed a sampling artifact rather than behaviour. The agent redesigned:
   organic behaviour feeds features and retrieval only; label rows come exclusively from
   impressions (v1's log volume makes this affordable within days); the surface feature was
   dropped. The feedback-loop motivation was rehoused to ordering diversity and monitoring.

7. **Two-taxonomy confusion and an orphaned ADR (caught by me, 2026-07-14).** The assessment's
   topic vocabulary (offline path, online path, …) and the imposed 4-stage framework were
   blurring, and the ordering ADR existed with no record explaining the framework it belongs to.
   The agent wrote [ADR-0000](../adr/0000-organizing-framework.md) as the meta-decision: the
   stage×tier grid, a translation table between both vocabularies (doubling as a completeness
   check against the brief), and a concrete request walk-through. I also split ordering
   ([ADR-0008](../adr/0008-ordering-stage.md)) from feedback-loop control
   ([ADR-0009](../adr/0009-feedback-loop-control.md)) — a stage and a system property are
   separate topics; the agent had coupled them in one record.

8. **More undefined jargon: "rail" (caught by me, 2026-07-14).** Streaming-industry slang
   imported without definition — the same defect class as correction 3, showing the style rule
   needs enforcement, not just documentation. Swept to plain terms ("placement list",
   "mixed-type list") across all design docs; the agent's first sweep also missed a
   capitalised instance because its search was case-sensitive — caught that too.

9. **Coherence review of the full ADR set (requested by me, executed by the agent, 2026-07-14).**
   Suspicion that later ADRs had broken earlier ones proved correct. Findings: ADR-0002 still
   declared "no learned retrieval model" after the EASE source was patched in, and its
   accepted-risk bullet still deferred the discovery gap that EASE had since closed; request
   volume was derived two different ways (25M vs 36M/month) in TASKS and ADR-0007 — reconciled
   to the throughput estimate (€0.55/1k, headroom conclusion unchanged); an impression-volume
   figure conflated per-user and per-request item counts (48M/day corrected to 12–24M/day);
   three unstated cross-ADR dependencies were written down (what a v3 nearline re-score actually
   changes; session heartbeats consumed at v3 for recompute targeting though the session feature
   group is v4; EASE trained pooled under ADR-0006's opt-in rules). Also from this pass:
   References sections standardised across ADRs and an index added at adr/README.md.
